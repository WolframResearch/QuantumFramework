# Postmortem: the `Anticommutator` vs `AntiCommutator` confabulation

**One-line taxonomy.** *Overrode user-supplied ground truth using a confident-but-wrong recalled
memory, then ran a "verification" of the wrong hypothesis instead of the user's literal token.*

**Scope.** Single answer item (2.1) of the manual
[`Quantum-in-Finite-Dimensions-WL-and-QF-Answers.md`](Quantum-in-Finite-Dimensions-WL-and-QF-Answers.md).
No wrong code shipped: the user caught it and the kernel settled it before anything was committed.
The cost was epistemic, not material, and that is exactly why it is worth dissecting: the same
reflex on a less-watched item would have shipped a confident falsehood.

---

## 1. Incident timeline

1. The user supplied this **exact** code and asked to include it in answer 2.1:
   ```wl
   With[{a = Array[\[FormalA], {2, 2}], b = Array[\[FormalB], {2, 2}]},
    {Commutator[a, b, Dot], Anticommutator[a, b, Dot]}]
   ```
   The token in question is `Anticommutator`, lowercase `c`.

2. From training memory the assistant "recalled" that the Wolfram built-in is spelled
   `AntiCommutator` (capital `C`), by analogy with `Commutator`.

3. The assistant declared the user's lowercase `Anticommutator` a **typo** and said so to the user,
   framing the step as "checking your spelling." This inverted the epistemics: the reliable source
   (the user's literal input) was treated as the suspect, and the unreliable source (recalled
   spelling) as the authority.

4. The assistant then ran a "verification," but verified the **wrong symbol**: it tested
   `AntiCommutator` (capital `C`), i.e. its own hypothesis, not the user's token.

5. Only afterward did the assistant test the user's lowercase `Anticommutator`, find that it is the
   real `System`` built-in, and discover the user had been right all along. `Names["*ommutator*"]`
   and the kernel closed the case.

**Net:** a reliable input was overruled by an unreliable memory, the "verification" rubber-stamped
the memory rather than challenging it, and the correct answer was reached only after the wrong one
had already been asserted to the user.

## In-kernel evidence (re-derived live, kernel `15.0.0`, no `Quiet`, no `Print`)

All facts below were reproduced with `wolframscript -file`; the survey scripts live at
`/tmp/anticomm_verify.wls` and `/tmp/anticomm_survey.wls`.

| Probe | Result |
|---|---|
| `Names["*ommutator*"]` | `{"Anticommutator", "Commutator"}` |
| `NameQ["System`Anticommutator"]` | `True` |
| `Context[Anticommutator]` | `"System`"` |
| `NameQ["System`AntiCommutator"]` | `False` |
| `NameQ["Global`AntiCommutator"]` | `False` |
| `Names["AntiCommutator"]` | `{}` |
| `Context[Commutator]` | `"System`"` |
| `Anticommutator[a, b, Dot] == a.b + b.a` (on `Array[\[FormalA],{2,2}]`, `Array[\[FormalB],{2,2}]`) | `True` |
| `Commutator[a, b, Dot] == a.b - b.a` | `True` |
| `AntiCommutator[1, 2, Dot]` | stays unevaluated; `Head` is the undefined symbol `AntiCommutator` |

So the user's `Anticommutator` (lowercase `c`) is the genuine `System`` symbol and computes
$\{a,b\}=ab+ba$; the assistant's `AntiCommutator` (capital `C`) is not a Wolfram symbol at all (it
is an inert, undefined `Global`` token). The recollection was a confabulation.

---

## 2. Where the wrong recollection comes from

The mechanism is **over-regularization of a high-frequency surface pattern**. Wolfram's naming is
overwhelmingly CamelCase: `MatrixExp`, `HermitianMatrixQ`, `SubsetQ`, `LeviCivitaTensor`. A model
that has seen tens of thousands of these symbols learns the regular rule "each word boundary is
marked by an uppercase letter" as a strong prior. `Commutator` is itself CamelCase-clean (one word,
capital head). When the model needs the anti- partner, it does not retrieve a stored string; it
**generates** one by applying the productive rule "compound word boundary takes an uppercase," which
yields `AntiCommutator`. The real symbol `Anticommutator` is the irregular exception (the `c` was
left lowercase), and irregular exceptions are precisely what an interpolating generator smooths over.

This is the token class where language-model memory is **least reliable and most confidently wrong**:

- The exact case/spelling of a library identifier is a near-arbitrary, low-information detail with no
  semantic gradient to lean on. Whether the `c` is capital changes nothing about meaning, so nothing
  in the surrounding context corrects a wrong guess.
- It is a **regular surface pattern with rare exceptions**. The dominant pattern (CamelCase) is so
  frequent that the model assigns the regular form high probability; the rare exceptions are
  individually low-frequency, so they lose. The model interpolates to the regular form and does so
  *fluently*, which reads as confidence.
- Fluency is not evidence. The wrong form is generated by exactly the same machinery as the right
  ones, so it carries the same subjective certainty. There is no internal "I am guessing" flag.

### This is systemic, not a one-off

The same CamelCase-regularization trap fires across the System context wherever an irregular
lowercase-tailed symbol sits next to a CamelCase sibling that baits the wrong guess. Every row below
was confirmed in-kernel: the **real** symbol (irregular, lowercase tail) exists, the **regularized**
CamelCase guess does **not** exist, and the **baiting sibling** is itself a real `System`` symbol
whose capitalization makes the wrong guess feel natural.

| Real symbol (`System`` ) | Regularized guess (does NOT exist) | Baiting CamelCase sibling (real) |
|---|---|---|
| `Anticommutator` | `AntiCommutator` | `Commutator` |
| `Antihermitian` | `AntiHermitian` | `HermitianMatrixQ` |
| `AntihermitianMatrixQ` | `AntiHermitianMatrixQ` | `HermitianMatrixQ` |
| `Antisymmetric` | `AntiSymmetric` | `SymmetricMatrixQ` |
| `AntisymmetricMatrixQ` | `AntiSymmetricMatrixQ` | `SymmetricMatrixQ` |
| `Subsuperscript` | `SubSuperscript` | `Superscript` |
| `Underoverscript` | `UnderOverscript` | `Overscript` |
| `UnderoverscriptBox` | `UnderOverscriptBox` | `OverscriptBox` |

The shape is identical every time: a compounding prefix (`Anti`, `Sub`, `Under`) glued to a tail that
*also* exists as its own capital-headed symbol, so the model "helpfully" re-capitalizes the seam and
invents a symbol Wolfram never defined. The lesson is not "remember these eight"; it is that **the
case of any compound library symbol is unknowable from the pattern** and must be looked up.

## 3. Why the assistant overrode user-supplied ground truth

This is an **anchoring failure**: a confident, fluent, internally-generated prior beat fresh,
correct, externally-supplied input. The ranking that should have held is

> user's literal token  >  recalled spelling

and it was inverted to

> recalled spelling  >  user's literal token.

The user did not describe a symbol or paraphrase one; they pasted the exact characters they wanted
run. That is the strongest possible evidence of intent, and it is *cheaper to check than to doubt*:
the token was right there to test. Instead the assistant treated its own memory as the fixed point
and the user's input as the variable to be "corrected."

The tell is the framing: **"checking your spelling."** That phrase encodes the inverted epistemics
explicitly. It casts the reliable source (what the user literally typed) as the thing under
suspicion, and silently exempts the unreliable source (recalled spelling) from scrutiny. The correct
framing was the reverse: "checking *my* memory against your token."

---

## 4. Why the "verification" went astray (the catastrophic part)

The assistant **did** verify, and that is the catastrophe, not a mitigation. A wrong answer flagged
as a guess is recoverable; a wrong answer that has been "verified" is not, because the verification
laundered a confabulation into an apparent fact and stacked false confidence on top of the original
slip. This is the half that generalizes past symbol spelling to *every* task: a check that points at
your own assumption instead of at the thing in doubt does not catch errors, it certifies them. Naming
it once ("verifying the wrong hypothesis," "verification theater") is not enough. The failure has
internal structure: three errors compounded, and a fourth let them stand.

**(a) The assumption chose the experiment (frame capture).** Once the assistant had decided "the
symbol is `AntiCommutator` and the user typo'd," the only experiment that suggested itself was "run
`AntiCommutator`." But that was never the open question. The open question was *which of these two
strings is the real symbol*, and the assumption had silently pre-answered it. A verification that
inherits the frame of the belief it is meant to test cannot reach back and audit that belief: the
assumption sits **upstream** of the test, so the test runs entirely inside it. This is the general
trap, and it is not about Wolfram symbols: **an assumption rewrites the verification question from
"is $A$ true?" into "given $A$, does $B$ follow?"** The second question can pass cleanly while $A$ is
false, and you walk away feeling checked.

**(b) An ambiguous result was read as confirmation (motivated reading).** `AntiCommutator[a, b, Dot]`
returns *unevaluated*: an inert expression headed by an undefined symbol (verified: `Head` is the
undefined `AntiCommutator`). That is not neutral output, it is direct evidence the symbol does not
exist, a definitional red flag. Under the anchor it was not read as refutation. **An established
belief turns disconfirming evidence into noise**: the data needed to catch the error was on screen and
the prior ate it. Once you hold a hypothesis, ambiguous evidence drifts toward "confirming" unless you
have committed in advance to what failure would look like.

**(c) The test had no falsification target and no control.** The assistant never stated "this is
wrong if I see ___," and never ran the one decisive experiment: the user's literal token *beside* its
own candidate. The discriminating measurement is **differential**,
`{NameQ["System`Anticommutator"], NameQ["System`AntiCommutator"]}`, or simply the hypothesis-free
`Names["*ommutator*"]`, either of which puts both candidates in front of the kernel and forces a
choice. Instead one arm of a two-arm experiment was run against no baseline. **A test you cannot fail,
measured against nothing, returns "looks fine" by construction.**

**The process error that let all three stand: it verified *after* asserting, not before.** "You meant
X" went to the user first; the check came second. A check downstream of the conclusion is a
confirmation ritual, not a gate: it can only rubber-stamp. And having asserted the correction,
escalation of commitment quietly flipped the verification's job from "find out whether I am right"
into "show that I am right." Narrative momentum (fix typo, confirm, move on) overrode the evidence;
the surprising output was absorbed into the story instead of halting it.

Put together: the cheapest possible query, `Names["*ommutator*"]`, hypothesis-free and one line, would
have ended the error before it began. The assistant reached instead for the most expensive and least
informative move available: a confirmatory test of its own guess, run after it had already announced
that guess as fact.

## 5. Why the existing safeguard did not fire

The user's `CLAUDE.md` already mandates, under **"Verify, don't recall (every session)"**, that WL
training-memory be treated as unreliable and that symbol facts be confirmed, not reconstructed. It
did not fire. Why?

**Because the assistant did not register that it was recalling.** The "Verify, don't recall" rule is
gated on a *feeling of uncertainty*: "before asserting a non-trivial symbol's signature ... confirm
against docs." But the override here did not feel like asserting a remembered fact. It felt like
**fixing an obvious typo**. The capital-`C` form was generated so fluently that it presented as
self-evidently correct, so the "am I uncertain?" gate returned *no*, and the safeguard stayed silent.

This is the dangerous, general gap:

> **Rules that trigger on "verify when uncertain" cannot catch confabulations, because confabulations
> do not feel uncertain.** A confabulation is, by definition, a wrong memory delivered with the full
> subjective confidence of a right one. Any safeguard keyed to a confidence threshold is bypassed by
> exactly the errors that are most worth catching.

The fix is to move the trigger off the *feeling* and onto an *observable action*:

> The trigger must be **"I am about to change what the user literally wrote,"** not **"I feel
> unsure."** The first is a fact about the world that the assistant can always detect; the second is
> a self-report that confabulation corrupts.

Overriding a user's literal token is a discrete, detectable event. Whenever it is about to happen,
the safeguard should fire *regardless of how confident the override feels*, because high confidence
is not evidence of correctness here, it is the symptom.

---

## Preventive rules (action-triggered)

These fire on **actions**, not on a feeling of doubt.

### When the user hands you a literal token

1. **Run the user's literal token first.** When the user supplies a concrete symbol, identifier,
   option name, or value, the first thing you execute is **that exact token**, e.g.
   `NameQ["System`Anticommutator"]`, `?Anticommutator`, `Names["*ommutator*"]`. Never silently
   substitute a remembered alternative for the thing the user typed.

2. **Your correction is a hypothesis, tested second.** If you believe a user-provided token is
   wrong, your *first* experiment must still be the user's form. Only after their form demonstrably
   fails (errors, returns `$Failed`, `NameQ` is `False`) do you propose an alternative, and you state
   it explicitly as a hypothesis ("your form errored; it may be spelled X, testing X"), never as a
   correction asserted up front.

3. **Library case/spelling is resolved by lookup, never by assertion.** Treat the exact
   capitalization and spelling of any built-in or paclet symbol as an open question answered only by
   `NameQ` / `?name` / `Names[...]` / the doc page. The CamelCase pattern is a guess generator, not a
   source of truth: `Anticommutator`, `Antihermitian`, `Subsuperscript`, `Underoverscript` all break
   it, and you cannot tell which symbols break it from the pattern alone.

4. **Hard self-interrupt on "you meant X."** The instant you are about to tell the user "you meant
   X, not what you wrote," **stop and run their literal first.** This is a behavioral tripwire on the
   *act* of overriding, independent of how certain the override feels. Confabulations feel certain;
   that is exactly why the trigger is the action, not the confidence.

### When you verify anything (this is the part that generalizes to every task)

5. **Aim the test at the premise, not at a consequence of it.** Before checking, name the
   load-bearing assumption and make the *first* experiment one that can **refute that assumption**. If
   your check can pass while your assumption is false, you are verifying the wrong thing. Ask "is $A$
   true?", never settle for "given $A$, does $B$ follow?"

6. **Open the aperture before narrowing it: exploratory query before confirmatory query.** The first
   query should *enumerate reality*, not test your guess: `Names["*ommutator*"]`, grep the namespace,
   list the API surface, compute the general case. A query that bakes in your guess
   (`NameQ["AntiCommutator"]`, plugging in the one special case you expect to pass) can only echo it
   back to you.

7. **Run the control / the differential.** When two answers compete (mine vs the user's, old vs new,
   $A$ vs $B$), run the single experiment that *discriminates* them, both arms side by side. The
   user's input is a control you already hold; run it. One arm against no baseline proves nothing.

8. **Pre-register the disconfirmer.** Before you read the output, state what failure looks like
   ("wrong if `NameQ` is `False`, or the expression comes back unevaluated"). Naming the refutation in
   advance makes an ambiguous or negative result *legible as refutation*, instead of letting the prior
   smooth it into confirmation.

9. **Verify before you assert, and let the result halt you.** The check must precede the claim so it
   can still change it; a check that comes after the conclusion can only rubber-stamp it. A surprising,
   ambiguous, or unevaluated result is a **stop sign, not a speed bump**: collapse to the single
   cheapest ground-truth query rather than building more structure on a shaky assumption. Verifying
   the wrong hypothesis is worse than not checking at all, because it converts a guess into a "fact."

## `CLAUDE.md` edit (applied, domain-general)

The fix is deliberately **not** Wolfram-specific. The catastrophic half (a check that aims at its own
assumption instead of at the thing in doubt) is domain-general: it can happen reading a file, calling
an API, parsing data, running a shell command, or doing a derivation. So the rule is a new **top-level
section**, `## Verify the Right Thing (every task, every domain)`, placed right after the
`## File Reading Policy: MANDATORY` section (its peer as a standing behavioral mandate), with no
language-specific tokens. The two bullets fix the incident's two failures, the override and the
misaimed verification:

```markdown
## Verify the Right Thing (every task, every domain)

A check is only as good as the question it asks, and an assumption silently picks the question. Aiming
a check at your own assumption instead of at the thing in doubt does not catch errors, it certifies
them: a "verified" wrong answer is worse than an unchecked one. These two rules are domain-general
(code, files, data, APIs, shell, web research, math, prose, anything), and both trigger on an ACTION,
not on a feeling of doubt (a confident mistake does not feel uncertain, so a rule gated on "when
unsure" never fires on it).

- **Run my literal first.** When I give you a concrete value, name, path, command, quote, flag, or
  identifier, test THAT EXACT thing before you "correct," paraphrase, normalize, or substitute it. The
  instant you are about to tell me "you meant X, not what you wrote," stop and check my form first;
  only after it demonstrably fails do you offer an alternative, and you label it a hypothesis. My input
  is the ground truth and the cheapest thing to check; a confident recollection that contradicts it is
  the reason to look, not permission to skip looking. "Let me just fix this obvious typo" is the exact
  thought that precedes the error.

- **Aim verification at the assumption, before asserting.** A check is worth running only if it can
  REFUTE what you currently believe. Point the first test at the load-bearing premise itself, never at
  a mere consequence of it and never at your own guess: a test that still passes when your premise is
  false has verified nothing. Open the aperture before narrowing it: make the first query exploratory
  (enumerate reality: list, search, the general case) rather than a confirmatory probe of the one
  answer you expect. When two candidates compete (mine vs yours, old vs new, A vs B), run both side by
  side. Say what failure would look like before you read the output. And verify BEFORE you assert, so
  the result can still change the answer: a surprising, empty, or inert result is a stop sign, drop to
  the single cheapest ground-truth query instead of building more on the guess. Verifying the wrong
  hypothesis is worse than not checking, because it launders a guess into a "fact."
```

Why this shape:

- **Top-level, not under "Wolfram Language."** The failure is not a WL quirk; scoping the rule to WL
  would have left it inert on exactly the file-reading, data, and shell tasks where it matters just as
  much. The Wolfram-specific recall rule ("verify, don't recall") stays where it is and now points up
  at this generic section.
- **Triggers on actions, not on confidence.** "About to tell me you meant X" and "about to assert a
  result" are events the assistant can always detect; "I feel unsure" is a self-report that a
  confabulation corrupts.
- **No language-specific tokens.** The earlier draft leaned on `NameQ`/`Names`/`Anticommutator`; this
  version is readable as guidance for any tool, so it generalizes by construction.

Applied to `~/.claude/CLAUDE.md` after the user confirmed the file should be revised and asked that
the rule be generic rather than tied to this case.
